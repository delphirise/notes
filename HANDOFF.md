# Notes App — Handoff Guide

This document is intended for the next person taking ownership of the repo. It describes the structure, where things live, how the app works, and the most common tasks (editing content, running the app, building CSS, and troubleshooting).

---

## 🧩 High-Level Structure

### Project roots
- `index.html` — the landing shell that links to the various tool pages.
- `app/` — the core runtime content for each tool/page.
- `assets/` — shared static assets (logos, PDFs, vCards, etc.).
- `tooling/` — build scripts, tests, and local-serving helpers.
- `docs/` — documentation (this file lives here).

### Key folders inside `app/`
- `app/pages/` — full HTML pages (each tool is a self-contained page). Examples:
  - `letters.html` (the letter generator)
  - `SUD_Diagnostic_Tool.html`, `Templates.html`, etc.
- `app/src/` — shared JS used by multiple pages.
- `app/vendor/` — prebuilt third-party libraries (jsPDF, html2canvas, etc.).
- `app/dist/` — built Tailwind CSS output (generated by `tooling/npm run build`).

---

## 🚀 Running the App Locally

### 1) Install dependencies
```bash
npm --prefix tooling install
```

### 2) Start a local server (recommended)
```bash
npm --prefix tooling run serve
```

Then open:

- `http://localhost:8080/index.html`
- or `http://localhost:8080/app/pages/letters.html` (or any other tool page)

> ✅ Running under a local HTTP server avoids `file://` CORS issues and enables PDFs and remote resources to load properly.

## 🛠️ How the App Works

### Page structure
Most tools are implemented as a self-contained HTML page (e.g., `letters.html`). Each page typically has:

1. **HTML UI markup** — forms, inputs, buttons, output text areas.
2. **Embedded JS** — generation logic, saving settings, PDF export.
3. **Built CSS** — Tailwind output (via `app/dist/output.css`).

### Tailwind CSS
- Source CSS is in `app/src/input.css`.
- Build output is written to `app/dist/output.css`.
- Rebuild with:
  ```bash
  npm --prefix tooling run build
  ```

### Vendor libraries
These are stored in `app/vendor/` and included by pages via relative `<script src="../vendor/...">`.

Common libraries:
- `jspdf.umd.min.js` (PDF generation)
- `html2canvas.min.js` (renders DOM to image for PDF)
- `pdf-lib.min.js` (merges PDFs, appends referrals, etc.)
- `crypto-js.min.js` (used for password check in `letters.html`)

## ✍️ Editing `letters.html` (Letter Generator)

This is the most complex page. Key sections:

### ✅ Password gating
- `#password-overlay` blocks access until a correct password is provided.
- Password is validated in `checkPassword()`.
- Password is stored as an MD5 hash inside `letters.html`.
  - If you want to change the password, update the hash (or remove the gate entirely).

### ✅ Templates & Forms
Each letter type is a `<div id="...Fields">` section which is toggled by `handleTemplateChange()`.
- Examples:
  - `generalUseFields` (General use letter)
  - `dischargeFields`
  - `enrollmentVerificationFields`

If you add a new template:
1. Add a new `<div id="<id>Fields" class="hidden form-section">` section.
2. Add an option to the `<select id="letterType">` control.
3. Create a `generate<YourType>Report()` function and include it in `generateReport()`.

### ✅ Report generation logic
- `generateReport()` calls the correct generator based on the selected template.
- Each generator builds `generatedText` and then `appendResourcesToTextarea()` adds the appendix.
- The PDF export is handled via `downloadPDF()`.

### ✅ Signature persistence
- Signature fields and the image are stored in `localStorage` under key `delphiRiseSignatureInfo`.
- `saveSignature()` / `loadSignature()` / `deleteSignature()` manage this.

There are scripted tests in `tooling/tests/`.

### Run all smoke checks
```bash
npm --prefix tooling run smoke
```


## ✅ Common Maintenance Tasks

### ✅ Update Tailwind styles
1. Edit `app/src/input.css`.
2. Run:
   ```bash
   npm --prefix tooling run build
   ```

### ✅ Add/modify a tool page
1. Add/edit `app/pages/<YourPage>.html`.
2. Ensure you include:
   - `<link href="../dist/output.css" rel="stylesheet">`
   - `<script src="../vendor/...">` scripts if needed
3. Optional: add navigation to `index.html`.

### ✅ Add a new vendor library
1. Drop the script file into `app/vendor/`.
2. Update the page(s) that need it by referencing `../vendor/<file>`.

## 🔍 Where to Look When Things Break

### ❌ CSS looks wrong / missing styling
- Ensure `app/dist/output.css` exists (run `npm --prefix tooling run build`).
- Verify pages reference it via `../dist/output.css`.

### ❌ Page won’t unlock / password won’t work
- Open `app/pages/letters.html` and check `checkPassword()`.
- The correct password is `delphi` (hash can be updated as needed).

### ❌ PDF download failing / missing assets
- The app tries to combine:
  - page-generated PDF (via jsPDF/html2canvas)
  - `Referrals.pdf` appended via pdf-lib
- If `Referrals.pdf` is missing or blocked by CORS, you can point it at the local `assets/documents/Referrals.pdf` or place it in the same folder as the HTML.

---

## 👍 Tips for New Maintainers
- Prefer editing structured HTML/JS inside `app/pages/*.html` rather than inlining behavior into a classless pile of code; keep generators grouped.
- Re-run smoke tests after changes to avoid breaking browser-rendered flows.
- When editing `letters.html`, keep the `generate*Report()` functions in sync with the fields (IDs must match).
